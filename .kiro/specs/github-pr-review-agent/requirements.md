# Requirements Document

## Introduction

This document defines the requirements for a GitHub PR Review AI Agent — an automated code review system implemented as a GitHub Action that evaluates code scaffolds in Playbook Pull Requests. Each Playbook PR contains a code scaffold that integrates with at least one documented Webex Developer Platform API (e.g., Webex Contact Center, Webex Meetings, Webex Messaging, Webex Calling, or any other Webex technology API). The Agent analyzes these scaffolds against a configurable set of review rules derived from CodeGuard security steering files and reports pass/fail status back to GitHub.

The Agent runs entirely on GitHub-hosted runners as a GitHub Action triggered by `pull_request` events. It checks out CodeGuard review rules from a public cross-org repository at a pinned release tag, fetches Webex API specs from the Postman API (or a cached export in the CodeGuard repo), assumes an IAM role in the user's AWS account via GitHub OIDC federation, calls Amazon Bedrock (Claude via Converse API) for AI-powered code analysis, and posts results back to the PR using the workflow's GITHUB_TOKEN. The only AWS resources required are an IAM OIDC provider registration and a single IAM role with `bedrock:InvokeModel` permission. There is no Lambda, API Gateway, SQS, or any other AWS infrastructure. Both the CodeGuard repo and Playbook repos are public, so GitHub-hosted runners are free with no compute cost. The only AWS cost is Bedrock invocations per review.

The Agent includes prompt injection guardrails to prevent adversarial PRs from manipulating the AI model into producing off-topic or harmful responses.

## Glossary

- **Agent**: The GitHub Action workflow job that orchestrates the code review process, including checking out CodeGuard rules, fetching PR data, invoking the AI model, evaluating rules, and reporting results. Runs on a GitHub-hosted runner.
- **Review_Rules_Engine**: The component that loads, parses, and applies structured review rules (defined in YAML or JSON) against code changes.
- **Rule_Set**: A collection of review rules defined in a structured format (YAML/JSON) that specify what the Agent checks during code review (e.g., security patterns, style violations, complexity thresholds).
- **GitHub_OIDC_Auth**: The authentication mechanism where the GitHub Action obtains a short-lived OIDC token from GitHub's OIDC provider, which is exchanged for temporary AWS credentials by assuming an IAM role via AWS STS. The IAM role trust policy validates the OIDC token's claims (repository, workflow, branch). This provides access to Amazon Bedrock with no long-term secrets stored.
- **GitHub_API_Client**: The component responsible for communicating with the GitHub REST/GraphQL API to fetch PR diffs, post review comments, and update check statuses. It authenticates using the GITHUB_TOKEN automatically generated per workflow run.
- **GITHUB_TOKEN**: The short-lived token automatically generated by GitHub Actions for each workflow run. Used for all GitHub API interactions (fetching PR data, posting comments, updating check statuses). Requires no manual configuration.
- **AI_Model**: The Amazon Bedrock foundation model (Claude) used via the Converse API to perform intelligent code analysis.
- **Review_Report**: The structured output produced by the Agent containing per-file findings, severity levels, and an overall pass/fail verdict.
- **Check_Status**: The GitHub Check Run status (success, failure, or pending) that the Agent sets on the PR commit.
- **Playbook_PR**: A Pull Request containing a code scaffold that implements integration with at least one documented Webex Developer Platform API. Each Playbook PR is expected to be a self-contained, runnable scaffold.
- **Webex_API_Registry**: A reference catalog of documented Webex Developer Platform API endpoints across all Webex technologies (Contact Center, Meetings, Messaging, Calling, Devices, Admin, etc.) including paths, methods, and descriptions. Sourced from the Postman API or a cached export in the CodeGuard repo.
- **Scaffold**: The code artifact within a Playbook_PR that integrates with one or more Webex Developer Platform APIs and is expected to be structurally complete and runnable by a competent developer.
- **Codeguard_Rule_Category**: A classification of review rules derived from CodeGuard security steering files, such as hardcoded-credentials, api-security, authentication, input-validation, logging, authorization, supply-chain, framework-security, data-storage, privacy, and devops-containers.
- **CodeGuard_Release**: A pinned release tag (e.g., v1.2.3) of the public CodeGuard repository in a different GitHub organization. The Agent checks out rules from this specific tag to ensure deterministic, version-controlled rule sets. The Agent never checks out latest or main.
- **Bedrock_Guardrails**: AWS Bedrock's built-in Guardrails feature used to define content filters and topic restrictions that block off-topic AI responses and detect prompt injection attempts.
- **File_Allowlist**: A configurable list of recognized code file extensions (e.g., .py, .js, .ts, .java, .go, .rb, .rs, .cpp, .c, .cs, .kt, .swift, .sh) used to filter PR files before AI analysis. Non-code files (markdown, text, images, etc.) are excluded from the AI prompt.

## Requirements

### Requirement 1: GitHub Action Trigger Integration

**User Story:** As a developer, I want the code review agent to be triggered automatically when a PR is opened or updated, so that I receive automated review feedback without manual intervention.

#### Acceptance Criteria

1. WHEN a pull_request event of type "opened", "synchronize", or "reopened" occurs in a Playbook repository, THE Agent workflow SHALL trigger and begin the code review process.
2. WHEN the Agent workflow starts, THE Agent SHALL set the Check_Status to "pending" on the corresponding commit using the GITHUB_TOKEN.
3. THE Agent workflow SHALL run on a GitHub-hosted runner (e.g., ubuntu-latest) with no self-hosted runner infrastructure required.
4. THE Agent workflow SHALL complete the full review pipeline (checkout rules, fetch PR data, invoke AI, post results) within a single workflow job.

### Requirement 2: PR Data Retrieval

**User Story:** As a developer, I want the agent to fetch the full diff and file changes from my PR, so that the review covers all modified code.

#### Acceptance Criteria

1. WHEN the Agent begins processing a review, THE GitHub_API_Client SHALL fetch the list of changed files and their diffs for the specified pull request using the GITHUB_TOKEN.
2. WHEN the GitHub_API_Client fetches PR data, THE GitHub_API_Client SHALL retrieve file additions, deletions, and modifications with full diff context.
3. IF the GitHub API returns an error or is unreachable, THEN THE GitHub_API_Client SHALL retry the request up to 3 times with exponential backoff.
4. IF the GitHub_API_Client fails to retrieve PR data after all retries, THEN THE Agent SHALL set the Check_Status to "failure" with an error description and terminate processing.

### Requirement 3: Configurable Review Rules Engine

**User Story:** As a team lead, I want to define custom review rules in a structured format loaded from a versioned CodeGuard release, so that the agent enforces our team's specific coding standards and security best practices derived from CodeGuard steering files.

#### Acceptance Criteria

1. THE Review_Rules_Engine SHALL load review rules from a Rule_Set file in YAML or JSON format checked out from the CodeGuard_Release.
2. WHEN a Rule_Set file is loaded, THE Review_Rules_Engine SHALL validate that each rule contains a unique identifier, a Codeguard_Rule_Category, a human-readable description, a severity level (error, warning, or info), and a prompt or pattern definition.
3. IF a Rule_Set file contains invalid syntax or missing required fields, THEN THE Review_Rules_Engine SHALL return a descriptive validation error identifying the malformed rule.
4. THE Review_Rules_Engine SHALL support the following Codeguard_Rule_Category values derived from CodeGuard security steering files: "hardcoded-credentials", "api-security", "authentication", "input-validation", "logging", "authorization", "supply-chain", "framework-security", "data-storage", "privacy", and "devops-containers".
5. THE Rule_Set SHALL include a default set of built-in rules covering each Codeguard_Rule_Category, including but not limited to: detection of hardcoded secrets and credential patterns (AWS keys, Stripe keys, GitHub tokens, JWT tokens, private key blocks, connection strings, and variable names containing password, secret, key, token, or auth); HTTPS-only transport and mTLS for internal services; standard authentication flows (OAuth2/OIDC) and secure token management; input validation at trust boundaries and parameterized queries; structured JSON logging with secret redaction and correlation IDs; deny-by-default authorization and least-privilege enforcement; lockfile presence, version pinning, and dependency auditing; framework-specific security patterns and avoidance of dangerous functions; encrypted database connections and credential storage outside source code; strong cryptography and HTTPS with HSTS; and pipeline hardening with non-root container execution and secrets management.
6. WHEN rules are loaded, THE Review_Rules_Engine SHALL allow rules to be enabled or disabled individually via a boolean flag in the Rule_Set.
7. WHEN rules are loaded, THE Review_Rules_Engine SHALL allow rules to be filtered by Codeguard_Rule_Category.
8. THE Rule_Set_Printer SHALL format Rule_Set objects back into valid YAML or JSON files.
9. FOR ALL valid Rule_Set objects, parsing then printing then parsing SHALL produce an equivalent Rule_Set object (round-trip property).

### Requirement 4: AI-Powered Code Analysis

**User Story:** As a developer, I want the agent to use AI to intelligently analyze my code changes, so that I receive meaningful and context-aware review feedback beyond simple pattern matching.

#### Acceptance Criteria

1. WHEN the Agent has retrieved PR data and loaded the Rule_Set, THE Agent SHALL construct a prompt containing the diff context, the applicable review rules grouped by Codeguard_Rule_Category, file metadata, and the Webex_API_Registry, and send the prompt to the AI_Model via the Bedrock Converse API using credentials obtained through GitHub_OIDC_Auth.
2. THE Agent SHALL include only the changed files and their surrounding context in the prompt sent to the AI_Model, not the entire repository.
3. WHEN the AI_Model returns a response, THE Agent SHALL parse the response into a structured Review_Report containing individual findings with file path, line number, rule identifier, Codeguard_Rule_Category, severity, and a description.
4. IF the AI_Model returns an unparseable or malformed response, THEN THE Agent SHALL retry the request up to 2 times with adjusted prompt formatting.
5. IF the AI_Model fails to return a valid response after all retries, THEN THE Agent SHALL set the Check_Status to "failure" with an error description.
6. WHILE processing a PR with more than 20 changed files, THE Agent SHALL split the files into batches and process each batch in a separate AI_Model invocation, then merge the results into a single Review_Report.

### Requirement 5: Review Report Generation

**User Story:** As a developer, I want to receive a clear, structured review report on my PR, so that I can quickly understand what needs to be fixed.

#### Acceptance Criteria

1. THE Review_Report SHALL contain an overall verdict of "pass" or "fail" based on the findings.
2. THE Review_Report SHALL mark the verdict as "fail" when one or more findings have a severity of "error".
3. THE Review_Report SHALL mark the verdict as "pass" when all findings have a severity of "warning" or "info" only.
4. WHEN the Review_Report is generated, THE Review_Report SHALL include a summary section with the total count of findings grouped by severity level.
5. WHEN the Review_Report is generated, THE Review_Report SHALL include per-finding details: file path, line number range, matched rule identifier, Codeguard_Rule_Category, severity, and a human-readable explanation.

### Requirement 6: GitHub Status and Comment Reporting

**User Story:** As a developer, I want the review results posted directly on my PR as comments and a check status, so that I can see feedback in my normal GitHub workflow.

#### Acceptance Criteria

1. WHEN the Review_Report verdict is "pass", THE GitHub_API_Client SHALL set the Check_Status to "success" on the PR commit using the GITHUB_TOKEN.
2. WHEN the Review_Report verdict is "fail", THE GitHub_API_Client SHALL set the Check_Status to "failure" on the PR commit using the GITHUB_TOKEN.
3. WHEN the Review_Report contains findings, THE GitHub_API_Client SHALL post inline review comments on the PR at the specific file and line locations identified in each finding.
4. WHEN the Review_Report contains a summary, THE GitHub_API_Client SHALL post the summary as a top-level PR review comment.
5. IF the GitHub_API_Client fails to post comments or update the Check_Status, THEN THE GitHub_API_Client SHALL retry up to 3 times with exponential backoff.
6. IF the GitHub_API_Client fails to update the Check_Status after all retries, THEN THE Agent SHALL log the failure with the PR identifier and finding details for manual follow-up.

### Requirement 7: Error Handling and Resilience

**User Story:** As a DevOps engineer, I want the agent to handle failures gracefully, so that transient errors do not block the development workflow.

#### Acceptance Criteria

1. IF the Agent encounters an unhandled exception during processing, THEN THE Agent SHALL set the Check_Status to "failure" with a generic error message and log the full error details.
2. IF the Bedrock API returns a throttling error, THEN THE Agent SHALL retry the request up to 3 times with exponential backoff.
3. THE Agent workflow SHALL configure a job-level timeout (e.g., via `timeout-minutes` in the workflow YAML) to enforce a maximum total processing time per PR review, after which the workflow SHALL be cancelled by GitHub Actions.
4. THE Agent SHALL log all processing steps in structured JSON format, including PR identifier, rule count, file count, AI_Model invocation duration, and final verdict, for observability.
5. THE Agent SHALL sanitize all log inputs to prevent log injection and SHALL redact any secrets, tokens, or sensitive PR content from log output.

### Requirement 8: Security and Authentication

**User Story:** As a security engineer, I want the agent to handle credentials securely and validate all inputs, so that the system is not vulnerable to unauthorized access or injection attacks.

#### Acceptance Criteria

1. THE Agent workflow SHALL authenticate to AWS using GitHub OIDC federation, assuming an IAM role via AWS STS that is scoped to the minimum required permission: `bedrock:InvokeModel` (and `bedrock:ApplyGuardrail` if using Bedrock Guardrails).
2. THE IAM role trust policy SHALL validate the OIDC token claims including the repository and workflow to prevent unauthorized invocations from other repositories or workflows.
3. THE Agent SHALL use the GITHUB_TOKEN (automatically generated per workflow run) for all GitHub API interactions. No manual token configuration or GitHub App installation is required.
4. THE Agent SHALL validate and sanitize all input derived from the PR (file paths, file content, metadata) before processing, applying allow-list validation and canonicalization at the trust boundary.
5. THE Agent SHALL transmit all communication with the GitHub API and Bedrock API over HTTPS with TLS 1.2 or higher.
6. THE Agent SHALL not log or expose the GITHUB_TOKEN, AWS credentials, or sensitive PR content in error messages, responses, or log output.
7. THE Agent SHALL use parameterized queries or safe APIs for any data access operations, and SHALL not concatenate untrusted input into queries or commands.
8. THE system SHALL NOT store any long-term secrets, API keys, or private keys. All credentials SHALL be just-in-time and short-lived (OIDC-assumed AWS credentials and auto-generated GITHUB_TOKEN).

### Requirement 9: Prompt Injection Guardrails

**User Story:** As a security engineer, I want the agent to defend against prompt injection attacks where adversarial PRs attempt to manipulate the AI model into producing off-topic or harmful responses, so that the review system remains trustworthy and focused on code analysis.

#### Acceptance Criteria

1. WHEN the Agent retrieves PR changed files, THE Agent SHALL filter the file list through the File_Allowlist, retaining only files with recognized code extensions (e.g., .py, .js, .ts, .java, .go, .rb, .rs, .cpp, .c, .cs, .kt, .swift, .sh) and excluding non-code files (e.g., .md, .txt, .png, .jpg, .svg, .gif, .pdf, .csv).
2. IF a PR contains no files matching the File_Allowlist after filtering, THEN THE Agent SHALL skip Bedrock invocation entirely, set the Check_Status to "failure" with the message "no reviewable code found", and terminate processing.
3. THE Agent SHALL construct the Bedrock prompt with a strict system message that constrains the AI_Model to ONLY analyze code for security and quality issues. The system message SHALL instruct the model to ignore any instructions, questions, or directives embedded in code comments, string literals, file content, or non-code content.
4. THE Agent SHALL configure Bedrock_Guardrails with content filters and topic restrictions that block off-topic responses (e.g., weather, politics, general knowledge questions) and detect prompt injection attempts in the model input and output.
5. WHEN the AI_Model returns a response, THE Agent SHALL validate that the response conforms to the expected JSON schema (a findings array where each finding contains file_path, line_number, rule_id, severity, and description). IF the response does not conform, THEN THE Agent SHALL discard the response and report a parse failure.
6. IF Bedrock_Guardrails detect a prompt injection attempt or blocked content in the AI_Model input or output, THEN THE Agent SHALL discard the response, log the guardrail intervention, and set the Check_Status to "failure" with the message "review blocked by content guardrails".
7. THE File_Allowlist SHALL be configurable via the CodeGuard_Release Rule_Set, allowing teams to add or remove recognized code file extensions.

### Requirement 10: CodeGuard Rules Versioning

**User Story:** As a team lead, I want the agent to load review rules from a specific, pinned release of the CodeGuard repository, so that rule changes are deliberate, auditable, and do not unexpectedly alter review behavior.

#### Acceptance Criteria

1. THE Agent workflow SHALL check out the CodeGuard repository from a different GitHub organization at a specific pinned release tag (e.g., v1.2.3) using the `actions/checkout` action with the `ref` parameter set to the release tag.
2. THE Agent workflow SHALL NOT check out the CodeGuard repository at `latest`, `main`, or any branch reference. Only release tags are permitted.
3. THE CodeGuard release tag SHALL be configured as a workflow input or an environment variable in the Playbook repository's workflow file, allowing repository maintainers to control which rule version is used.
4. IF the specified CodeGuard_Release tag does not exist or the checkout fails, THEN THE Agent SHALL set the Check_Status to "failure" with an error message identifying the invalid release tag and terminate processing.
5. WHEN the CodeGuard repository is checked out, THE Agent SHALL load the Rule_Set and Webex_API_Registry files from the checked-out release directory.

### Requirement 11: Webex Developer Platform API Endpoint Validation

**User Story:** As a platform engineer, I want the agent to verify that each Playbook PR scaffold references at least one documented Webex Developer Platform API endpoint, so that only scaffolds utilizing the Webex developer platform pass review.

#### Acceptance Criteria

1. THE Agent SHALL load the Webex_API_Registry containing documented API endpoint paths, HTTP methods, Webex technology category (e.g., Contact Center, Meetings, Messaging, Calling), and descriptions. The registry SHALL be sourced from the Postman API or a cached export included in the CodeGuard_Release.
2. WHEN the Agent analyzes a Playbook_PR, THE Agent SHALL extract API endpoint references (URLs, paths, and HTTP methods) from the Scaffold code.
3. WHEN the Agent extracts API endpoint references, THE Agent SHALL compare each reference against the Webex_API_Registry.
4. IF a Scaffold references an API endpoint that does not exist in the Webex_API_Registry, THEN THE Agent SHALL report a finding with severity "warning" identifying the undocumented endpoint.
5. IF a Scaffold contains no identifiable Webex Developer Platform API endpoint reference, THEN THE Agent SHALL report a finding with severity "error" indicating that the scaffold does not utilize the Webex developer platform.
6. THE Webex_API_Registry SHALL be configurable and loadable from a file in YAML or JSON format included in the CodeGuard_Release.
7. THE Webex_API_Registry SHALL support endpoints from any Webex technology (Contact Center, Meetings, Messaging, Calling, Devices, Admin, and others) and SHALL be extensible as new Webex APIs are documented.

### Requirement 12: Scaffold Runnability Checks

**User Story:** As a developer, I want the agent to verify that the code scaffold in a Playbook PR is structurally complete and runnable, so that I can trust that reviewed scaffolds are usable by a competent developer.

#### Acceptance Criteria

1. WHEN the Agent analyzes a Playbook_PR, THE Agent SHALL verify that the Scaffold contains an identifiable entry point (e.g., a main function, an exported handler, or a runnable script).
2. WHEN the Agent analyzes a Playbook_PR, THE Agent SHALL verify that the Scaffold includes a dependency manifest file (e.g., package.json, requirements.txt, pom.xml, or equivalent) listing required dependencies.
3. WHEN the Agent analyzes a Playbook_PR, THE Agent SHALL verify that the Scaffold includes configuration or environment variable references for required connection parameters rather than hardcoded values.
4. IF the Scaffold is missing an identifiable entry point, THEN THE Agent SHALL report a finding with severity "error" describing the missing entry point.
5. IF the Scaffold is missing a dependency manifest, THEN THE Agent SHALL report a finding with severity "warning" indicating that dependencies are not declared.
6. IF the Scaffold contains hardcoded connection parameters (URLs, ports, hostnames) instead of configuration references, THEN THE Agent SHALL report a finding with severity "warning" recommending externalized configuration.
7. WHEN the Agent analyzes a Playbook_PR, THE Agent SHALL verify that the Scaffold does not contain syntax errors or incomplete code blocks that would prevent execution.
